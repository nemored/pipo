# Specification: Elixir-Supervised Rust Transports (Single Binary, Tarball Distribution)

## 1. Purpose

Define the target architecture, interfaces, packaging, and delivery gates for moving supervision to Elixir while retaining existing Rust transport implementations. This specification is intended to be directly convertible into an execution plan with concrete engineering tasks.

## 2. Scope

### In scope
- Introduce an Elixir OTP supervisor that manages Rust transport workers via Ports.
- Keep transport behavior changes minimal; limit Rust changes primarily to runtime API/entrypoint and stdio protocol adaptation.
- Produce a relocatable tarball release containing:
  - Elixir release runtime
  - one static Rust transport binary (`pipo-transport`)
  - config examples
  - artifact manifest/checksums
- Add test coverage prior to implementation for protocol, supervision behavior, and packaging.

### Out of scope
- Rewriting transport protocol/client logic (Slack/Discord/IRC/Mumble/Rachni internals).
- New transport features unrelated to supervision/runtime integration.
- Replacing existing business logic for message transformation.

## 3. Repository Layout (Target)

```text
/
├─ apps/
│  └─ pipo_supervisor/               # Elixir OTP app (Supervisor + Port workers)
├─ native/
│  └─ transport_runtime/             # Rust crate/workspace for pipo-transport
```

## 4. Runtime Architecture

### 4.1 Process Model
- Elixir app starts as primary runtime and owns lifecycle.
- Each configured transport is represented as one supervised child worker.
- Every worker spawns the same executable (`pipo-transport`) with a transport selector argument.
- The Router GenServer is listed first in the supervision tree and starts before any transport worker. Workers must tolerate a brief window where the router is unavailable, failing publishes gracefully rather than crashing.

Example worker launch:
- `pipo-transport --transport slack --config /path/to/config.json`

### 4.2 Supervision Semantics
- Strategy: `one_for_one`.
- Child restart policy defaults to permanent unless explicitly configured.
- Restart intensity and window are configurable (e.g., max N restarts in T seconds).
- Backoff with jitter is applied to repeated failures.
- Graceful shutdown flow:
  1. send protocol `shutdown` message;
  2. wait `shutdown_timeout_ms`;
  3. force terminate process if still alive.
- The Router GenServer can crash and restart independently without affecting transport worker processes.

### 4.3 Message Routing Model
- Rust workers communicate only through stdin/stdout protocol.
- All cross-transport message routing is owned by the Router GenServer (see 4.4).
- Existing bus behavior is preserved functionally (fanout by logical bus name).

### 4.4 Router Design

#### Routing table
The Router builds its routing table from config at startup — not from runtime messages. The existing channel_mapping in each transport definition (mapping transport-specific channel names to bus names) is inverted at load time to produce a bus-to-worker index:

```
bus "general" → [:slack_0, :irc_0, :discord_0]
bus "ops"     → [:slack_0, :irc_0]
```

The table is fully populated before any worker is started. When a worker emits a `ready` frame, the Router cross-checks its declared `transport` and `instance_id` against the expected table and rejects anything unrecognised. No dynamic subscription negotiation is required.

#### Router state
```elixir
%{
  routing_table: %{
    "general" => [:slack_0, :irc_0, :discord_0],
    "ops"     => [:slack_0, :irc_0]
  },
  worker_pids: %{slack_0: #PID<...>, irc_0: #PID<...>},
  next_pipo_id: 1
}
```

#### pipo_id ownership
The Router owns pipo_id assignment. It assigns a pipo_id before fanout and stamps it onto all outbound `inject_message` frames. This replaces the previous `Arc<Mutex<i64>>` shared counter in the Rust runtime; no cross-process locking is needed.

#### Self-send filtering
Self-send filtering moves entirely to the Router. When worker `:slack_0` publishes to bus `"general"`, the Router looks up the subscriber list, removes the originating worker, and delivers only to the remaining subscribers. Rust transports no longer perform self-send filtering. The `sender` field on outbound `message` frames is retained for logging and debugging purposes only.

#### Backpressure
- Workers publish to the Router via `GenServer.cast` (fire-and-forget). The Router's mailbox is the buffer.
- The Router delivers to workers via `GenServer.call` with a configurable short timeout (default 200ms).
- If a worker does not ack within the timeout, the Router logs a structured warning, marks that worker `degraded`, and drops the message for that worker only — delivery to other workers is unaffected.
- If a worker accumulates N consecutive dropped deliveries (configurable), the Router notifies the supervisor, which applies the configured restart policy.

This ensures a slow or crashed worker is a local failure rather than a global stall. Missed messages during a degraded window are acceptable for a chat bridge and should be documented in operator documentation.

#### Dynamic transport management
The Router is designed to support adding and removing transports at runtime. The routing table should be treated as mutable state from the outset. Hardening of dynamic add/remove operations is deferred to a later phase, but the data structures must not preclude it.

## 5. Rust Binary Contract (`pipo-transport`)

### 5.1 CLI
Required flags:
- `--transport <irc|discord|slack|mumble|rachni>`
- `--config <path>`

Optional flags:
- `--instance-id <string>`
- `--protocol-version <v1>` (defaults to current supported)

Exit codes:

| Code | Meaning |
|------|---------|
| 0    | Orderly shutdown |
| 2    | Invalid CLI args or config parse failure |
| 3    | Protocol version incompatibility |
| 10   | Transport authentication failure |
| 11   | Transport network unreachable at startup |
| 12   | Transport-specific fatal error (details logged before exit) |
| 20   | Internal runtime panic |

Note: the supervisor's restart policy may key on specific exit codes. In particular, exit code 10 (auth failure) should not trigger indefinite retries; operators should be notified and retries held until config is updated.

### 5.2 Startup Sequence
1. Parse args and config section for selected transport.
2. Initialize transport runtime.
3. Emit `ready` frame with:
   - protocol version
   - transport type
   - instance id
4. Enter event loop (read stdin commands, process transport IO, write stdout events).

The supervisor enforces a `ready_timeout_ms` window. If no `ready` frame is received within this window the worker is marked `degraded` and the normal restart policy applies.

### 5.3 Protocol Framing
- UTF-8 newline-delimited JSON (NDJSON).
- One object per line.
- Unknown `type` values are ignored with a `warn` log.
- All timestamps are RFC3339 strings, UTC only, millisecond precision. Format: `"2026-03-13T12:00:00.000Z"`. Neither side may emit local-offset timestamps. In Rust: `chrono::Utc::now().to_rfc3339_opts(SecondsFormat::Millis, true)`. In Elixir: `DateTime.utc_now() |> DateTime.to_iso8601()`.

### 5.4 Protocol Messages

Inbound (Elixir → Rust):
- `shutdown`
- `inject_message`
- `health_check`
- `reload_config` — **reserved**; not implemented in phase 1. If received, the transport must emit an `unimplemented` `warn` frame and continue operating. This reservation ensures phase 2 implementation does not require a protocol version bump.

Outbound (Rust → Elixir):
- `ready`
- `message`
- `health`
- `warn`
- `fatal`

All frames include:
- `type`
- `protocol_version`
- `transport`
- `instance_id`
- `timestamp` (RFC3339, UTC, millisecond precision — see 5.3)

#### `message` frame (Rust → Elixir)
Must include a `bus` field identifying the logical bus this message belongs to. The `pipo_id` field is omitted or null — the Router assigns it during fanout.

```json
{
  "type": "message",
  "bus": "general",
  "transport": "slack",
  "instance_id": "slack_0",
  "protocol_version": "v1",
  "timestamp": "2026-03-13T12:00:00.000Z",
  "pipo_id": null,
  "payload": {}
}
```

#### `inject_message` frame (Elixir → Rust)
Includes `pipo_id` (assigned by Router) and `origin_transport` (for logging; replaces the former in-process self-send filter field).

```json
{
  "type": "inject_message",
  "bus": "general",
  "pipo_id": 4217,
  "origin_transport": "slack",
  "protocol_version": "v1",
  "timestamp": "2026-03-13T12:00:00.000Z",
  "payload": {}
}
```

## 6. Elixir Worker Contract

Each worker module must:
- Resolve executable path (env override first, release-relative fallback).
- Start a Port with binary mode and line-oriented packets.
- Decode JSON frames and route to the Router GenServer.
- Handle non-zero exit status with restart semantics from supervisor.
- Track child health state (`starting`, `ready`, `degraded`, `stopped`).
- Tolerate Router unavailability at publish time by logging and dropping rather than crashing.

## 7. Configuration

### 7.1 Supervisor Config
- `protocol_version`
- `shutdown_timeout_ms`
- `ready_timeout_ms` — maximum time to wait for a `ready` frame from a worker before marking it `degraded` and applying restart policy
- `require_all_ready` — if `true`, supervisor treats any worker that fails to reach `ready` within `ready_timeout_ms` as a fatal boot error; defaults to `false`. Recommended value is `true` for CI smoke tests.
- `restart_intensity_max_restarts`
- `restart_intensity_window_seconds`
- `backoff_min_ms`, `backoff_max_ms`
- `router_delivery_timeout_ms` — timeout for Router-to-worker `call` delivery; defaults to 200ms
- `router_degraded_drop_threshold` — consecutive dropped deliveries before Router notifies supervisor
- transport definitions (type, enabled, transport-specific config path)

### 7.2 Transport Config
- Existing transport settings retained.
- Must support selecting one transport block by `--transport`.

## 9. Quality Gates (Tests First)

Implementation may start only after these tests are added and wired in CI.

### 9.1 Protocol Tests
- `ready` handshake contains required fields including RFC3339 timestamp.
- version mismatch yields reject path.
- malformed JSON line handling does not crash supervisor process.
- `reload_config` receipt by Rust worker emits `warn` frame with type `unimplemented` and worker continues operating.

### 9.2 Supervision Tests
- crashing child restarts under policy.
- restart intensity trips correctly and escalates as configured.
- graceful shutdown timeout causes forced termination.
- worker marked `degraded` after exceeding `router_degraded_drop_threshold`.
- Router crash and restart does not affect transport worker processes.

### 9.3 Runtime Selection Tests
- `pipo-transport --transport <name>` selects correct runtime.
- invalid transport value exits with code 2.
- auth failure exits with code 10.

### 9.4 Packaging Tests
- tarball contains required files.
- binaries are executable post-extract.
- smoke test boots supervisor with `require_all_ready: true` and receives at least one `ready` from a worker.

## 10. CI/CD Requirements

Minimum required pipeline jobs:
1. Rust lint/test/build (`pipo-transport`)
2. Elixir lint/test/build (`mix release`)
3. Integration tests (protocol + supervision)
4. Tarball assembly
5. Extract-and-smoke-test tarball

Release publish is gated on all above passing.

## 11. Implementation Phases and Task Backlog

### Phase 0 — Branch + Test Scaffolding
1. Create branch `feat/elixir-supervision-tarball-single-binary`.
2. Add protocol fixture set under `schema/fixtures/` including timestamp format examples and `reload_config` reserved frame fixture.
3. Add failing tests for sections 9.1–9.4.
4. Wire test jobs in CI as required checks.

### Phase 1 — Rust Runtime Adaptation
1. Add `pipo-transport` CLI dispatcher.
2. Add stdio protocol module and frame schemas including `bus` field on `message` frames and `origin_transport`/`pipo_id` on `inject_message` frames.
3. Integrate existing transports behind transport selector. Remove shared `pipo_id` mutex and self-send filter from Rust transports; replace `channels: HashMap<channel_name, broadcast::Sender>` with `channels: HashMap<channel_name, bus_name>`.
4. Implement startup `ready` and shutdown handling.
5. Implement `reload_config` stub that emits `unimplemented` warn and continues.

### Phase 2 — Elixir Supervisor Integration
1. Add Router GenServer with routing table construction from config, pipo_id assignment, self-send filtering, and backpressure/drop policy.
2. Add Port worker module with `ready_timeout_ms` enforcement and Router-tolerant publish.
3. Add supervisor child spec with Router listed first, restart/backoff config, and `require_all_ready` boot flag.
4. Add health tracking and structured logging annotations (including `instance_id`, `transport`, restart counters).

### Phase 3 — Packaging and Release
1. Implement `packaging/build_tarball.sh`.
2. Generate manifest and checksums.
3. Implement `packaging/smoke_test.sh` with `require_all_ready: true`.
4. Add release documentation (`README-runtime.md`).

### Phase 4 — Hardening
1. Add chaos-style tests for repeated crash/restart.
2. Validate long-running stability and memory growth.
3. Verify compatibility matrix for supported target triples.
4. Harden dynamic transport add/remove operations in the Router.

## 12. Acceptance Criteria

The initiative is complete when all criteria below are met:
1. Supervisor can start at least two transport instances via Ports using the same `pipo-transport` executable.
2. On child crash, configured restart policy executes and is observable via logs/metrics.
3. `shutdown` handshake works, including timeout escalation behavior.
4. Tarball builds in CI, extracts cleanly, and passes smoke tests with `require_all_ready: true`.
5. Protocol version mismatch is detected and fails fast with actionable errors.
6. Existing transport functional behavior remains intact except for runtime orchestration differences.
7. Router crash and restart is survivable without restarting transport workers.
8. `reload_config` receipt by a Rust worker produces an `unimplemented` warn frame; worker continues operating.

## 13. Risks and Mitigations

- **Protocol drift** between Elixir and Rust:
  - Mitigation: shared schema fixtures + handshake version gating.
- **Restart storms** under bad credentials/network failures:
  - Mitigation: restart intensity limits + exponential backoff/jitter + exit code 10 suppressing automatic retry.
- **Operational debugging complexity** across runtimes:
  - Mitigation: structured logs include `instance_id`, `transport`, restart counters; NDJSON frames are human-readable on raw stdout.
- **Packaging regressions**:
  - Mitigation: CI smoke test from built tarball, not source tree.
- **Router becoming a delivery bottleneck**:
  - Mitigation: per-worker delivery timeout with drop-and-degrade rather than blocking; Router mailbox is the only shared buffer.

## 14. Open Decisions

The following decisions have been resolved:

1. **`reload_config` phase**: deferred to phase 2; reserved in protocol from phase 1 with `unimplemented` warn stub.
2. **Timestamp format**: RFC3339 strings, UTC only, millisecond precision (`"2026-03-13T12:00:00.000Z"`). Enforced by schema fixtures.
3. **Boot behavior on failed readiness**: start degraded by default (`require_all_ready: false`); `require_all_ready: true` available as a config flag and required for CI smoke tests.

The following decision remains open and must be resolved before Phase 3:

4. **Deployment target triples**: initial set of supported targets for tarball releases has not been specified.
